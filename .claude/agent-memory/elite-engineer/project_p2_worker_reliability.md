Phase-2 worker reliability hardening (applied to working tree, NOT committed — user manually tests before commit).

**HIGH-1 zombie account (the worst):** `monitorAndReplyToComments` was called OUTSIDE `checkAndAct`'s try in the old `runOneCycle`, so a throw there rejected `runOneCycle` → broke the `while(true)` in `runAccount` → browser closed in finally → account dead forever while process stayed alive (PM2 never restarts). Fix = two-altitude loop: `runAccountSession` owns the inner cycle loop with `runCycleGuarded` (never rejects, swallows + logs cycle throws, writes per-account cycle-boundary heartbeat); `runAccount` is the supervisor — catches whatever escapes a session and respawns with exponential-capped backoff (5s base, 5min cap). Clean exit gated on module `shuttingDown` flag so shutdown doesn't fight the supervisor.

**Why:** A single bad cycle must continue to next interval; an abnormal account exit must restart that account only, never the whole process, and never silently.

**HIGH-2 heartbeat timing:** control-plane (server/) treats `worker_state.last_heartbeat` older than ~90s as dead, but a cycle (FB nav+scroll) routinely exceeds 90s. So heartbeat-per-cycle would false-positive dead. Fix = process-liveness ticker at 25s (`HEARTBEAT_INTERVAL_MS`, well under 90s), unref'd, idempotent (only first account starts it). PLUS per-account cycle-boundary status writes. `db.heartbeat(status, detail)` is the existing accessor (db.js:291) — signature is (status, detail=null), writes last_heartbeat=now. It had ZERO callers before this fix.

**HIGH-3 orphan browsers:** per-account-browser design = N chromium trees. No signal handler → PM2 SIGTERM hard-kills node → N orphaned trees → VPS OOM over many start/stop cycles. Fix = module-scope `liveBrowsers` Set in loop.js (register on launch, unregister on normal close), `shutdownAllBrowsers()` closes all with per-browser `Promise.race` timeout (8s). index.js owns signal wiring (SIGTERM/SIGINT → gracefulShutdown(exit 0); uncaughtException → cleanup then exit(1) so PM2 restarts whole process). gracefulShutdown has its own 12s hard ceiling + force-exit. Ownership split: index.js owns process lifecycle, loop.js owns browser handles.

**MEDIUM-1 per-id seen flush:** `fb/monitor.js` batched `writeSeenComments` at post-loop end; interrupt mid-loop lost in-memory Set → re-like/re-reply/re-DM already-actioned commenters (ban-risk amplifier). Fix = new `core/state.js markCommentSeen(account, postUrl, commentId)` (single-id, idempotent via db.addSeenComment ON CONFLICT DO NOTHING), called immediately after acting (before the articleErr catch, so a comment that failed mid-act is NOT marked → retried next cycle). Kept `writeSeenComments` exported (test/state.test.js uses it).

**uncaughtException policy change:** was log-and-continue; now log + browser cleanup + exit(1) (Node docs: process state undefined after uncaughtException, running on = corrupted zombie). unhandledRejection unchanged (log + stay alive — one account's stray rejection must not kill the fleet).

**SHUTDOWN-REVIEW round 2 (post-gate, 2026-06-12):** independent reviewer found a real HIGH I missed + confirmed my self-flag.
- HIGH register-after-sweep ORPHAN (TOCTOU race): `shutdownAllBrowsers` snapshots-then-CLEARS liveBrowsers. A browser whose `await launchBrowser()` was parked when SIGTERM landed would register into the swept Set → orphaned. Fix = guard at the CHOKE POINT: `registerBrowser` now checks `shuttingDown` (synchronous, no await between check and add → window collapses to zero for ALL callers), refuses + closes the late browser, returns boolean. `runAccountSession` bails early on `!registerBrowser(browser)` (no cycles, no double-close — browser already closed by register). Lesson: snapshot-then-clear sweep + a registration path that doesn't re-check the swept-flag = orphan; guard the single entry point, not each call site.
- MEDIUM backoff-never-resets: backoffMs only doubled, so a long-healthy account inherited up-to-5min recovery after one transient death. Fix = `computeNextBackoff(currentMs, sessionDurationMs)` pure helper (exported, unit-tested): session ran >= HEALTHY_SESSION_MS (60s) → reset to base; else double capped at max. Applied to backoffMs BEFORE the sleep (subtle: original inline reset-then-sleep was correct; a refactor that slept-old-then-evolved would make healthy deaths still wait the grown backoff — re-derived the timing). 60s threshold cleanly separates flap (launch-then-crash) from transient (min check interval is 1m).
- MEDIUM closeDb-never-called: `db.closeDb` (db.js:73, exported db.js:436) had zero callers. Added `db.closeDb()` in index.js gracefulShutdown finally, AFTER shutdownAllBrowsers (which stops the ticker first → no heartbeat-after-close), wrapped in try (double-close safe). Calling an existing db.js accessor is allowed; did NOT edit db.js. Verified ordering safe via standalone script.
- Skipped LOW (cycle heartbeat last-writer-wins across N accounts) → deferred to P5.

**How to apply:** loop.js test seams = `deps.launchBrowser` (fake browser), `deps.once`/`deps.skipHeartbeat` (test control), `startHeartbeat({heartbeat, setInterval})` (fake clock + spy), `__resetForTest()` (clears module shuttingDown/timer/registry between tests). test/loop.test.js covers HIGH-1/2/3 deterministically (no real browser/network/timers). eslint shared config (eslint.config.js) lists setTimeout but NOT setInterval/clearInterval/clearTimeout — use `/* global setInterval, clearInterval */` comment per server/app.js convention, do NOT edit the shared config.
